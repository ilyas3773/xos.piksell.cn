# 客户端原型歧义与开发注意事项

## 说明

- 这份文档只记录客户端原型里最容易误判的地方。
- 原因主要是墨刀里存在透明点击层、重叠热点、同一位置覆盖多个跳转区域。
- `README.md` 与 `NAVIGATION_MAP.md` 里的跳转是自动提取结果，适合做基础参考，但以下问题需要人工纠偏。

## 1. 认证页存在明显热点重叠

涉及页面：

- 03《登录》
- 04《注册》
- 05《忘记密码》
- 06《登录-密码登录》

观察到的问题：

- 自动提取结果中，03 页的“登录”热点跳到了 04《注册》，这在业务上明显不合理。
- 06 页中“忘记密码？”与“登录”的跳转结果也疑似互相串位。
- 04 页中“注册”与返回登录的关系也可能被透明层覆盖。

建议前端实现：

- 不要机械照抄这几页的自动跳转结果。
- 按正常业务逻辑实现：
  - 登录成功 -> 首页
  - 忘记密码入口 -> 忘记密码页
  - 注册入口 -> 注册页
  - 注册完成 -> 登录页或首页
  - 密码登录/验证码登录只是在登录页内切换模式

## 2. 首页与我的页底部导航存在覆盖

涉及页面：

- 08《商城》
- 10《我的》
- 11《首页》

观察到的问题：

- 自动提取结果里，底部 Tab 有多个“商城”“我的”“会员”互跳异常。
- 某些热点标签与目标页不符合视觉直觉，说明底部区域存在叠层。

建议前端实现：

- 底部导航按标准 4 Tab 实现即可：
  - 首页
  - 商城
  - 购物车
  - 我的
- 会员不建议放到底部主 Tab，更像二级入口页。
- 页面间互跳优先按截图视觉和业务常识处理，不以自动提取异常跳转为准。

## 3. 发票管理与地址管理在“我的”页中疑似反向

涉及页面：

- 10《我的》
- 37《我的-发票管理》
- 38《我的-地址管理》

观察到的问题：

- 自动提取结果里，“地址管理”跳到了发票页，“发票管理”跳到了地址页。
- 这大概率是命中文字标签时被相邻透明热点干扰，不建议直接采用。

建议前端实现：

- 地址管理 -> 38《我的-地址管理》
- 发票管理 -> 37《我的-发票管理》

## 4. 商品详情页存在大面积透明区域热点

涉及页面：

- 07《首页-商品详情》
- 12《首页-商品详情 -下单》
- 39《首页-商品详情 -我要批发》
- 46《首页-商品详情 Copy 1》

观察到的问题：

- 自动提取到的热点标签有时会混入说明文字，例如“立即购买 / 存储至阴凉处，避免太阳直射”。
- 这是因为购买按钮附近存在大面积透明点击层，覆盖了多个文本区域。

建议前端实现：

- 只保留清晰业务动作：
  - 加入购物车
  - 立即购买
  - 我要批发
- 说明文字不要做成点击元素，除非原型截图有明确交互暗示。

## 5. 若页面只有 `(unlabeled hotspot)`，优先按页面视觉判断

常见于：

- 微信号登录页
- 购物车页
- 下单页
- 订单状态页
- 发票/地址管理页

建议处理规则：

1. 先看截图中最像按钮、卡片、底部操作条的区域
2. 再参考 `README.md` 中的热点坐标
3. 最后才参考自动提取出的目标页

## 6. 更适合前端实现的统一抽象

普通订单相关页面：

- 不要拆成 10 多个完全独立页面
- 用一个订单详情页承接多个状态
- 用状态字段控制按钮文案和信息块显示

OEM 相关页面：

- 不要为每个阶段重复造 UI
- 用统一时间线 + 内容区的布局
- 阶段差异通过字段块、附件区和操作按钮控制

商品详情相关页面：

- 建议一个详情页组件，通过商品 mock 数据切换
- 下单页与批发页作为详情页延伸流程

## 7. 对前端 AI 的最终约束

- 截图优先级高于自动跳转结果
- 业务常识优先级高于异常热点映射
- 对有歧义的跳转，先在代码里以注释标记 `prototype-ambiguous`
- 若后续你要接入真实后端，应把订单状态与 OEM 阶段做成枚举，不要写死在页面名里
